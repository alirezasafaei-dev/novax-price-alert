# گزارش نهایی بررسی، تحلیل و پیشنهادهای بهبود پروژه بات تلگرامی

## 1) هدف مستند
این مستند خروجی نهاییِ جمع‌بندی بررسی‌ها، تحلیل‌ها و پیشنهادهای اجرایی برای بهبود پروژه و بات است.  
هدف آن این است که وضعیت فعلی پروژه به‌صورت شفاف جمع‌بندی شود، ریسک‌های اصلی مشخص شوند، و یک مسیر واقعی، حرفه‌ای و قابل اجرا برای بهتر شدن محصول تعریف شود.

این مستند با نگاه محصولی، فنی، عملیاتی و تجربه کاربری نوشته شده و فرض اصلی آن این است که بات برای **استفاده واقعی و روزمره کاربران ایرانی در تلگرام** ساخته می‌شود؛ نه صرفاً یک MVP نمایشی یا ابزار آزمایشی.

---

## 2) جمع‌بندی اجرایی
برداشت اصلی از وضعیت فعلی پروژه این است که هسته محصول شکل گرفته و ارزش اولیه آن روشن است، اما برای تبدیل شدن به یک بات قابل اعتماد و مناسب استفاده روزمره، چند مسئله کلیدی باید با اولویت بالا حل شوند.

در وضعیت فعلی، مهم‌ترین نیازهای پروژه این‌ها هستند:

- شفاف‌سازی رفتار واقعی سیستم، به‌ویژه در مسیر ساخت هشدار
- هم‌ترازسازی مستندات با رفتار واقعی کد و runtime
- بهبود UX برای کاهش ابهام و افزایش اعتماد کاربر
- سخت‌سازی فنی سیستم هشدار از نظر دقت، تکرار، و پایداری
- بهبود نمایش قیمت برای بازار و کاربر ایرانی
- آماده‌سازی پروژه برای رشد مرحله بعدی محصول

نتیجه مهم این بررسی این است که پروژه فعلاً بیشتر از توسعه قابلیت‌های جدید، به **clarification, hardening, alignment** نیاز دارد.

---

## 3) تعریف درست محصول
این بات باید به‌عنوان یک ابزار واقعی و روزمره برای کاربران ایرانی تعریف شود.  
این یعنی محصول باید طوری طراحی شود که کاربر بتواند:

- قیمت‌ها را سریع و واضح بفهمد
- دارایی موردنظرش را بدون ابهام دنبال کند
- هشدارها را با اطمینان ثبت کند
- به صحت و کیفیت هشدارها اعتماد کند
- در محیط تلگرام، بدون پیچیدگی اضافه، تجربه‌ای روان و مفید داشته باشد

این پروژه نباید فقط «اطلاعات قیمت» نمایش دهد.  
هدف درست آن باید این باشد که به یک **دستیار ساده، قابل اعتماد و کاربردی برای رصد دارایی‌ها** تبدیل شود.

---

## 4) اصول پایه محصول برای بازار ایران

### 4.1 قیمت‌ها باید روز و واقعی باشند
اگر بات برای استفاده واقعی ساخته می‌شود، قیمت‌های آن باید تا حد ممکن به بازار واقعی ایران نزدیک باشند.  
نمایش قیمت‌های غیرهم‌راستا با بازار یا قیمت‌هایی که از نظر زمانی stale هستند، خیلی سریع اعتماد کاربر را از بین می‌برد.

بنابراین این اصل باید ثابت باشد:
- قیمت‌ها باید به‌روز باشند
- منبع قیمت باید مشخص و قابل اتکا باشد
- تبدیل‌ها باید با نرخ معتبر انجام شوند
- در صورت وجود تاخیر، خطا، یا عدم قطعیت، سیستم باید آن را مدیریت کند

### 4.2 واحد نمایش اصلی باید تومان باشد
برای کاربر ایرانی، تومان واحد اصلی فهم و تصمیم‌گیری است.  
استفاده از ریال به‌عنوان واحد اصلی تجربه را ضعیف می‌کند و استفاده صرف از دلار نیز برای محصولی که بر بازار ایران متمرکز است کافی نیست.

بنابراین:
- واحد اصلی نمایش قیمت برای کاربر نهایی باید **تومان** باشد
- در پیام‌های اصلی، هشدارها، تاییدها و خروجی‌های قیمتی، تومان مبنا باشد
- اگر نیاز به نمایش نرخ مرجع یا قیمت دلاری وجود دارد، باید در لایه دوم و به‌صورت کمکی ارائه شود، نه به‌عنوان نمایش اصلی

### 4.3 قیمت باید قابل فهم باشد، نه فقط صحیح
یکی از مهم‌ترین تصمیم‌های محصول این است که نمایش قیمت فقط از نظر عددی درست نباشد؛ باید از نظر ذهنی و عملی هم برای کاربر مفید باشد.

مثلاً اگر قیمت یک دارایی گران‌قیمت به‌شکل عددی بسیار بزرگ نمایش داده شود، ممکن است از نظر محاسباتی درست باشد اما از نظر تجربه کاربری مفید نباشد.  
کاربر باید بتواند سریع تشخیص دهد:
- این دارایی در چه بازه‌ای است
- نسبت به قبل چه تغییری کرده
- آیا برای او قابل خرید/فروش در مقیاس معمول است یا نه
- هشدار ثبت‌شده دقیقاً بر چه واحدی اعمال می‌شود

### 4.4 محصول باید با عرف واقعی بازار ایران هماهنگ باشد
نمایش قیمت، انتخاب واحد، فرمت هشدار، و متن‌های کاربری باید با رفتار واقعی کاربران ایرانی و عرف سایت‌ها، صرافی‌ها و ابزارهای رایج بازار ایران هماهنگ باشند.

این یعنی:
- نمایش‌ها باید آشنا و سریع‌الفهم باشند
- واحدها باید با منطق خرید و فروش واقعی سازگار باشند
- عددها نباید بیش‌ازحد درشت، بیش‌ازحد ریز، یا شبیه خروجی خام API باشند
- پیام‌ها باید کوتاه، روشن و قابل اعتماد باشند

## 5) سیاست پیشنهادی نمایش قیمت

### 5.1 واحد اصلی
واحد پیش‌فرض و اصلی برای کاربر باید **تومان** باشد.

### 5.2 فرمت عددی
قیمت‌ها باید:
- خوانا باشند
- جداکننده هزارگان داشته باشند
- در واحد مناسب نمایش داده شوند
- از اعشار غیرضروری پرهیز کنند
- در محیط تلگرام سریع فهمیده شوند

### 5.3 اصل «واحد مناسب»
هر دارایی باید در واحدی نمایش داده شود که برای کاربر ایرانی هم **قابل فهم** باشد و هم **متناسب با رفتار واقعی معامله**.

این اصل مهم است:
- نه واحد باید آن‌قدر درشت باشد که عملاً برای کاربر بی‌معنا شود
- نه آن‌قدر ریز باشد که عددها بی‌ارزش یا گیج‌کننده شوند

یعنی محصول باید برای دارایی‌های مختلف، نمایش مناسب انتخاب کند.

### 5.4 منطق نمایش برای دارایی‌های مختلف
برای دارایی‌های گران‌قیمت، صرف نمایش قیمت یک واحد کامل همیشه بهترین UX نیست.  
برای دارایی‌های ارزان‌قیمت نیز نمایش با اعشارهای طولانی تجربه ضعیفی ایجاد می‌کند.

بنابراین پیشنهاد حرفه‌ای این است که برای هر دارایی یا هر دسته از دارایی‌ها policy مشخص تعریف شود:
- واحد مرجع
- واحد نمایشی اصلی
- تعداد اعشار
- فرمت متنی
- رفتار در هشدارها
- رفتار در صفحه/پیام تایید

### 5.5 اصل قابلیت تصمیم‌گیری
نمایش قیمت باید طوری باشد که کاربر بتواند با کمترین اصطکاک:
- قیمت را بفهمد
- تغییرات را درک کند
- هشدار تنظیم کند
- مطمئن شود هشدار روی همان چیزی اعمال می‌شود که در ذهنش بوده

## 6) وضعیت فعلی پروژه

### 6.1 جمع‌بندی وضعیت
بر اساس بررسی‌های انجام‌شده، پروژه در مرحله‌ای قرار دارد که می‌توان آن را یک MVP فعال و امیدوارکننده دانست.  
هسته اصلی محصول ظاهراً شکل گرفته و موارد زیر در آن وجود دارد یا برای آن طراحی شده است:

- نمایش قیمت
- تعریف هشدار
- پردازش پس‌زمینه
- ساختار اولیه عملیات و استقرار
- مستندات متعدد برای محصول، معماری، عملیات و استقرار

این وضعیت یک مزیت مهم دارد:  
پروژه از مرحله ایده عبور کرده و حالا می‌توان روی **کیفیت، اعتماد، و قابلیت توسعه** آن متمرکز شد.

### 6.2 نیاز واقعی فعلی
نیاز اصلی پروژه در این مرحله، بازطراحی وسیع یا توسعه شتاب‌زده قابلیت‌های جدید نیست.  
نیاز اصلی این است که:

1. رفتار واقعی سیستم دقیقاً روشن شود
2. ابهام‌های کاربری حذف شوند
3. ریسک‌های هشدار کاهش یابد
4. کد و مستندات با هم هم‌راستا شوند
5. تجربه کاربر ایرانی در نمایش قیمت و دریافت هشدار بهتر شود

به‌صورت خلاصه:
پروژه الان بیشتر از هر چیز به **شفاف‌سازی، سخت‌سازی، و حرفه‌ای‌سازی** نیاز دارد.

---

## 7) مهم‌ترین مسئله فعلی
مهم‌ترین مسئله فعلی پروژه، ابهام در Alert Flow است.

این ابهام می‌تواند در چند نقطه رخ دهد:
- نحوه شروع فرایند ساخت هشدار
- نحوه انتخاب دارایی
- استفاده یا عدم استفاده از context قبلی
- نمایش یا عدم نمایش شفاف دارایی در مرحله تایید
- اعتبارسنجی ورودی کاربر
- نحوه ثبت نهایی و رفتار پس از ثبت

اگر این مسیر برای کاربر کاملاً روشن نباشد، حتی اگر سیستم از نظر فنی «کار کند»، تجربه محصولی ضعیف می‌شود.

چرا این موضوع بحرانی است؟
چون هشدار یکی از هسته‌ای‌ترین قابلیت‌های این محصول است.  
کاربر اگر روی هشدار اعتماد نکند، ارزش بات به‌طور مستقیم کاهش پیدا می‌کند.

---

## 8) نتیجه قسمت اول
قسمت اول این مستند یک چارچوب پایه روشن می‌کند:

- این بات باید برای استفاده واقعی کاربران ایرانی طراحی شود
- تومان باید واحد اصلی نمایش باشد
- قیمت باید هم واقعی باشد و هم در واحدی نمایش داده شود که برای کاربر قابل فهم و قابل استفاده باشد
- محصول باید با عرف واقعی بازار ایران هماهنگ باشد
- مهم‌ترین نیاز فعلی پروژه، شفاف‌سازی رفتار واقعی، بهبود تجربه هشدار، و افزایش قابلیت اعتماد است

بر این اساس، ادامه این مستند روی سه محور اصلی بنا می‌شود:

1. تحلیل دقیق ریسک‌ها و شکاف‌های فعلی
2. تعریف پیشنهادهای عملی برای UX، UI، محتوا، فنی و عملیات
3. ارائه مسیر اجرایی مرحله‌بندی‌شده برای بهبود پروژه
